Architecture Decision Records 8 ADRs

ADR-001 Headless Core + Lit Rendering Layer Separation

Status: Accepted

Context: Data grid libraries traditionally bundle state management with rendering, creating tight coupling. TanStack Table proved the headless pattern works, but developers still need default rendering.

Decision: Two separate layers: @phozart/phz-core (headless, zero DOM) and @phozart/phz-grid (Lit Web Components rendering). Core runs in Node.js, Web Workers, or browser.

Consequence: Framework adapters are thin wrappers. Core can be tested without browser. State logic is reusable across rendering targets.

ADR-002 DOM Virtualization Default, Canvas Enterprise

Status: Accepted

Context: DOM virtualization supports native browser features (selection, a11y, CSS); canvas enables extreme performance for millions of rows.

Decision: DOM virtualization as default (accessible, styleable). Canvas rendering as opt-in enterprise feature for high-performance scenarios.

Consequence: Community tier gets full accessibility. Enterprise users can switch to canvas for million-row datasets.

ADR-003 Semantic Shadow DOM for Accessible Virtualization

Status: Accepted

Context: Core differentiator. Every major grid breaks screen reader accessibility with virtualization. AG Grid, Handsontable, MUI DataGrid, PrimeNG all have critical issues. Sarah Higley (Microsoft) publicly criticized AG Grid's ARIA implementation.

Decision: Semantic Shadow DOM approach that maintains correct ARIA attributes, focus management, and screen reader context during virtualization.

Consequence: phz-grid becomes the first data grid to solve accessible virtualization correctly.

ADR-004 DuckDB-WASM + Apache Arrow Data Backbone

Status: Accepted

Context: JavaScript arrays are inefficient for large-scale client-side analytics: no indexing, poor cache locality, single-threaded, no columnar operations.

Decision: DuckDB-WASM for in-browser SQL analytics with Apache Arrow for zero-copy data transfer between the engine and grid.

Consequence: 100x+ performance improvement for large datasets. SQL-powered filtering, sorting, aggregation in the browser.

ADR-005 Yjs CRDTs for Real-Time Collaboration

Status: Accepted

Context: CRDTs provide mathematically guaranteed convergence without central coordination, unlike Operational Transform (Google Docs pre-2021).

Decision: Yjs CRDT library for real-time collaboration with WebSocket and WebRTC sync providers.

Consequence: Offline-first, peer-to-peer capable. Presence awareness and conflict-free merging built-in.

ADR-006 Three-Layer CSS Custom Property Theming

Status: Accepted

Context: Traditional approaches (inline styles, CSS classes) have significant drawbacks. Inline styles can't use pseudo-classes; CSS class approaches lead to 500+ classes.

Decision: Three-layer CSS custom property system: Brand Tokens → Semantic Tokens → Component Tokens.

Consequence: Full dark mode support, Forced Colors Mode compatibility, easy theming with ~50 variables instead of 500+ classes.

ADR-007 Schema-as-Contract for AI Integration

Status: Accepted

Context: LLMs with function calling need structured contracts. Unstructured prompts produce freeform text with no type safety or validation.

Decision: Grid state API generates JSON Schema 7 that serves as the contract between AI models and the grid. Supports OpenAI, Anthropic, Google providers.

Consequence: Natural language queries produce structured, validated actions. Schema drives AI integration, docs, and type generation.

ADR-008 ESM-Only, No CommonJS

Status: Accepted

Context: In 2026, all modern browsers, Node.js 12+, and build tools (Vite, Rollup, esbuild) support ESM natively. CommonJS prevents tree-shaking and static analysis.

Decision: ESM-only distribution. No CommonJS, no UMD, no dual publishing.

Consequence: Full tree-shaking support. Core bundle target <50 KB gzipped. Simpler build configuration.

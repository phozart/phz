System Overview

The phz-grid BI stack is a modular, headless system for building reports, dashboards, and analytics UIs. It separates computation (engine) from rendering (widgets) and configuration (definitions, admin). Everything is TypeScript, ESM-only, and framework-agnostic via Web Components (Lit v3).

UI Layer: grid-admin, engine-admin, grid-creator, criteria Admin panels, wizards, filter UI
Rendering Layer: widgets (20+ components) KPI cards, charts, gauges, dashboards
Computation Layer: engine (BIEngine facade) Aggregation, pivot, expressions, status, drill-through
Serialization Layer: definitions GridDefinition, stores, converters, Zod validation
Core Layer: core (headless grid engine) State, events, types, views, row model
Key principle: The engine is pure computation with zero DOM dependencies. It runs in Node.js, Web Workers, or the browser. Widgets are Lit Web Components that consume engine output. Admin panels are separate packages that configure engine artifacts.

Filter Integration (Option D — Pragmatic Hybrid)

Two entry points, one unified system:

ModeHowWhen
Inline reportConfig.criteriaConfig passed directly Simple cases, developer-defined filters
Registry FilterRegistry + FilterBindingStore Admin-managed, reusable filter definitions
Auto-Hydration Inline config auto-registers into registry on load Backward compatibility; engine handles automatically

Resolution order: Registry bindings (if exist) → Auto-hydrated inline → Empty. Divergence detection alerts if both inline and registry exist but differ.

How To: Build a Report

A report in phz-grid is a configured data view with columns, sorting, filtering, aggregation, conditional formatting, and optional drill-through navigation.

Create the BI Engine
Initialize the engine facade. All registries are created internally.
import { createBIEngine } from '@phozart/phz-engine';

const engine = createBIEngine({
  // Optional: pre-load data
  initialDataProducts: [myDataProduct],
  // Optional: custom compute backend (default: JS)
  // computeBackend: myDuckDBBackend,
});
Register Data Products
Data products define the schema and source of your data.
engine.dataProducts.register({
  id: dataProductId('sales'),
  name: 'Sales Data',
  fields: [
    { name: 'region', type: 'string' },
    { name: 'revenue', type: 'number' },
    { name: 'date', type: 'date' },
  ],
});
Create a Report Configuration
Define columns, sorting, filtering, and optional aggregation. Use reports.createBlank() for a starter config. 
const report = engine.reports.createBlank('Sales by Region');
report.dataProductId = dataProductId('sales');
report.columns = [
  { field: 'region', header: 'Region', visible: true },
  { field: 'revenue', header: 'Revenue', width: 150, visible: true },
  { field: 'date', header: 'Date', visible: true },
];
report.sort = { field: 'revenue', direction: 'desc' };
report.aggregation = {
  fields: [
    { field: 'revenue', functions: ['sum', 'avg'] },
  ],
};
report.conditionalFormatting = [
  {
    field: 'revenue',
    rules: [
      { condition: 'greaterThan', value: 100000,
        style: { backgroundColor: '#dcfce7' } },
    ],
  },
];

engine.reports.save(report);
Convert to Grid Config & Render
Use toGridConfig() to produce a config compatible with createGrid(). 
import { createGrid } from '@phozart/phz-core';

const gridConfig = engine.reports.toGridConfig(report);
// gridConfig = { columns, sort, filter, presentation }

const grid = createGrid({
  columns: gridConfig.columns,
  data: myRawData,
  initialState: {
    sort: gridConfig.sort,
    filter: gridConfig.filter,
  },
});

// Bind to <phz-grid> element
const el = document.querySelector('phz-grid');
el.gridApi = grid;
Add Filters (Optional)
Use the ReportService for managed filter state with auto-resolution. 
import { createReportService } from '@phozart/phz-engine';

const reportService = createReportService(engine, report.id);
const fields = reportService.getFields();       // SelectionFieldDef[]
const params = reportService.getFilterParams();  // GridFilterParams

// Apply filters programmatically
reportService.setValue('region', ['US', 'CA']);

// Subscribe to filter changes
reportService.subscribe((filterParams) => {
  // Re-query or re-filter data
  grid.setState({ filter: filterParams.criteria });
});

// Connect to criteria UI
const criteriaEl = document.querySelector('phz-selection-criteria');
criteriaEl.config = { fields };
criteriaEl.addEventListener('criteria-apply', (e) => {
  reportService.setValues(e.detail.context);
});
Persist with Definitions (Optional)
Serialize the report to a GridDefinition for storage/sharing. 
import { gridConfigToDefinition, exportDefinition,
  createLocalStorageStore } from '@phozart/phz-definitions';

const def = gridConfigToDefinition(gridConfig, {
  name: 'Sales by Region',
  description: 'Regional revenue report',
});

// Export as JSON
const json = exportDefinition(def);

// Or persist to store
const store = createLocalStorageStore();
store.save(def);

Dashboards use a 12-column CSS grid. Each widget has position (row, col) and size (rowSpan, colSpan). 
const dashboardConfig = {

  widgets: [
    {

      type: 'kpi-card',

      position: { row: 0, col: 0 },
      size: { colSpan: 3, rowSpan: 2 },

      cardStyle: 'expanded',
    },
    {
      id: widgetId('region-chart'),
      type: 'bar-chart',
      title: 'Revenue by Region',
      position: { row: 0, col: 3 },
      size: { colSpan: 9, rowSpan: 4 },
      dataProductId: dataProductId('sales'),
      metricField: 'revenue',
      dimension: 'region',
      rankOrder: 'desc',
    },
    {

Render with <phz-dashboard>
The dashboard component resolves widget types and renders them in a CSS Grid layout. 
import '@phozart/phz-widgets';
import { applyTheme, darkTheme } from '@phozart/phz-widgets';

const dashboard = document.querySelector('phz-dashboard');
dashboard.config = dashboardConfig;
dashboard.engine = engine;
dashboard.data = myRawData;

// Apply theme
applyTheme(dashboard, darkTheme);

// Listen for drill-through events
dashboard.addEventListener('drill-through', (e) => {
  const { source, xValue, value } = e.detail;
  // Navigate to detail report with context
});
Add Global Filters (Enhanced Dashboard)
Use EnhancedDashboardConfig for global filters that scope to specific widgets. 
import { upgradeDashboardConfig } from '@phozart/phz-engine';

const enhanced = upgradeDashboardConfig(dashboardConfig);

  {
    id: 'region-filter',
    label: 'Region',
    type: 'multi_select',
    field: 'region',
    options: ['US', 'EU', 'APAC'],
    widgetScope: ['region-chart', 'bottom-5'], // only affects these
  },
];
enhanced.theme = 'dark';

How To: Define KPIs

Register a KPI Definition
KPIs have targets, thresholds, direction, breakdowns, and optional alerts. 
engine.kpis.register({
  id: kpiId('total-revenue'),
  name: 'Total Revenue',
  description: 'Monthly recurring revenue across all products',
  category: 'Financial',
  target: 1000000,
  unit: 'currency',
  unitLabel: 'USD',
  direction: 'higher_is_better',
  thresholds: { ok: 90, warn: 70 }, // percentage of target
  deltaComparison: 'previous_period',
  dimensions: ['region', 'product'],
  breakdowns: [
    { id: 'us', label: 'United States', targetOverride: 500000 },
    { id: 'eu', label: 'Europe', targetOverride: 300000 },
    { id: 'apac', label: 'APAC', targetOverride: 200000 },
  ],
  sparkline: { enabled: true, periods: 6 },
  defaultCardStyle: 'expanded',
  dataSource: {
    scoreEndpoint: '/api/kpi/revenue/score',
    trendEndpoint: '/api/kpi/revenue/trend',
    breakdownEndpoint: '/api/kpi/revenue/breakdown',
  },
});
Compute Status & Delta
Use the status engine to classify KPI values against thresholds. 
const kpi = engine.kpis.get(kpiId('total-revenue'));

// Status classification
const status = engine.status.compute(850000, kpi);
// { status: 'ok'|'warn'|'crit'|'unknown', value, threshold, percentage }

// Delta computation
const delta = engine.status.computeDelta(850000, 780000, kpi);
// { value: 70000, percentage: 8.97, direction: 'improving', unit: 'USD' }

// Full score classification
const score = { kpiId: 'total-revenue', value: 850000, previousValue: 780000 };
const classified = engine.status.classify(score, kpi);
// { status, delta, breakdowns: [...] }
Register Metrics (Calculated Values)
Metrics are named calculations. They support simple, conditional, composite, and expression formulas. 
// Simple metric
engine.metrics.register({
  id: metricId('avg-deal-size'),
  name: 'Average Deal Size',
  dataProductId: dataProductId('sales'),
  formula: { type: 'simple', field: 'deal_value', aggregation: 'avg' },
});

// Composite metric (combines others)
engine.metrics.register({
  id: metricId('revenue-per-rep'),
  name: 'Revenue per Rep',
  dataProductId: dataProductId('sales'),
  formula: {
    type: 'composite',
    expression: '{total-revenue} / {rep-count}',
    fields: [metricId('total-revenue'), metricId('rep-count')],
  },
});
Scorecard & Status Table
Display multiple KPIs at once with scorecard or status table widgets. 
// Scorecard widget in dashboard
{

Filter Integration

Architecture: Two Systems, One API

System A: Inline Config
criteriaConfig on ReportConfig or DashboardConfig. Developer writes filter definitions directly. Simple, code-first approach. 
report.criteriaConfig = {
  fields: [
    { id: 'region', label: 'Region',
      type: 'multi_select',
      options: [...] },
    { id: 'date', label: 'Date Range',
      type: 'date_range' },
  ],
};
System B: Filter Registry
Admin-managed, reusable definitions stored in FilterRegistry and bound to artefacts via FilterBindingStore. Enterprise-grade. 
// Register reusable definition
engine.criteria.registry.register({
  id: 'region-filter',
  label: 'Region',
  type: 'multi_select',
  dataField: 'region',
  options: [...],
});
// Bind to report
engine.criteria.bindings.bind({
  filterDefinitionId: 'region-filter',
  artefactId: reportArtefactId(report.id),
  order: 1,
  visible: true,
});

SaveController Pattern

Reusable Lit ReactiveController for managing async save state across admin components. 
import { SaveController } from '@phozart/phz-engine-admin';

// States: 'idle' -> 'saving' -> 'saved' (auto-dismiss 3s) -> 'idle'
//                            -> 'error' (manual dismiss/retry)

class MyAdmin extends LitElement {
  private save = new SaveController(this);

  async handleSave() {
    await this.save.save(async () => {
      await api.persist(this.getConfig());
    });
  }

  render() {
    return html`
      <phz-save-indicator .state=${this.save.state}></phz-save-indicator>
      <button @click=${this.handleSave}
              ?disabled=${this.save.state === 'saving'}>Save</button>
    `;
  }
}

Implementation Patterns

Facade Pattern
createBIEngine() returns a single object wrapping all subsystems (registries, compute, status, criteria). Always start here.
Pure Function + Component Split
Logic exported as pure functions (testable, reusable). Components import and call logic, handle rendering/events. Example: computeStackedSegments() (pure) → PhzBarChart (component).
Config Layering
Configs support 3 layers: system → admin → user. Use mergeReportConfigs(layers) or mergeDashboardConfigs(layers) to produce the effective config.
Widget Resolution Pipeline
resolveWidgetProps(widget, context) transforms widget config into render-ready props. Handles KPI scoring, chart projection, data filtering. Pure function.
5-Layer Expression DAG
Fields → Parameters → Calculated Fields → Metrics → KPIs. Circular dependency detection via DFS. Compiled closures are 5-10x faster than tree-walk.
Event Emission Pattern
All Lit components use dispatchEvent(new CustomEvent('name', { bubbles: true, composed: true, detail })). Parent listens and re-dispatches or handles.
Pluggable Compute Backend
Default is JSComputeBackend (pure JS). Replace with DuckDBComputeBackend for SQL push-down or a custom server backend. Interface: aggregate(), pivot(), filter(), computeCalculatedFields().
Serialization Round-Trip
gridConfigToDefinition(config, meta) → exportDefinition(def) → JSON string → importDefinition(json) → definitionToGridConfig(def). Auto-migrates schema versions.

Common Pitfalls & Guidelines

Do NOT import engine types in core. Core is the foundation; engine depends on core. Never create circular dependencies. If grid needs engine features, use dynamic import().
Always use branded IDs. Use kpiId('x'), reportId('x'), dashboardId('x'), etc. Do not pass raw strings where branded types are expected.
Widget components need loading/error handling. Every widget accepts loading: boolean and error: string | null. Always set these. The widget renders spinner/error/empty states automatically via resolveWidgetState().
PitfallSolution
Passing raw data to widgets without engine processing Use processWidgetData() or resolveWidgetProps() to transform data first
Forgetting .open binding on admin modals Always bind .open=${showAdmin} on <phz-grid-admin>, <phz-grid-creator>
Using innerHTML for widget content All widgets use Shadow DOM. Set properties via JS, not attributes for complex data
Not registering KPIs before using scorecard/status widgets Always engine.kpis.register() before rendering KPI widgets
Mixing inline criteriaConfig with registry when both exist Pick one. If both exist, registry takes precedence. Divergence callback fires.
Missing dataProductId on chart widgets Bar chart, trend-line, bottom-N all require dataProductId for data resolution
Not applying theme before rendering Call applyTheme() on the dashboard element before setting config. Themes cascade via CSS variables.
Stale filter state after report switch Call reportService.destroy() and create new service when switching reports
Expression evaluation returns null unexpectedly NULL propagates (SQL-style). Check inputs with IS NULL / COALESCE in expressions.
Dashboard layout overflow Widget colSpan must not exceed layout.columns (default 12). Use clampColSpan() from responsive-layout.

Build Order

When building packages, respect the dependency chain: definitions → core → engine → widgets → criteria → grid-admin → engine-admin → grid-creator. Always npm run build --workspace=packages/<pkg> after changes.

Testing

All computation is testable without DOM. Engine, aggregation, pivot, expression, status, and widget resolution are pure functions. Use Vitest directly.

Widget components test logic via exported pure functions (e.g., computeStackedSegments(), valueToAngle(), computeHeatmapCells()). DOM rendering tests use Lit's test utilities.

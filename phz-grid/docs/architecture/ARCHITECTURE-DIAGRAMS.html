Grid + Grid Configuration

How <phz-grid> renders data and how <phz-grid-admin> configures it via ReportPresentation.

Key Insight

  • grid-admin does NOT import from grid. It imports ReportPresentation types from engine.
  • The host application is responsible for wiring admin output to the grid's properties.
  • Grid re-exports everything from core, so consumers only need @phozart/phz-grid.

Report & Dashboard Filter Binding

How FilterBinding links reusable FilterDefinitions to reports and dashboards via ArtefactId.

Key Points

  • Same filter definition (e.g. "region") can be bound to multiple artefacts with different overrides (label, default, visibility).
  • ArtefactId is a branded string with prefix: report:sales-q4, dashboard:ops-main.
  • Bindings are the many-to-many join table between definitions and artefacts.
  • targetScope on bindings can further restrict to specific widgets within a dashboard.

BI Engine Facade

The createBIEngine() facade wires together all engine sub-systems.

BIEngine is a single facade

  • Created via createBIEngine(config). All sub-registries and engines are internal.
  • Reports and Dashboards both store optional criteriaConfig for inline filter definitions.
  • resolve-criteria.ts auto-hydrates inline CriteriaConfig into the registry if no bindings exist (backward compat).
  • report-service.ts is the runtime glue that subscribes to filter changes and produces GridFilterParams.

Dashboard + Widgets

How <phz-dashboard> renders widgets, and how the engine-admin configures them.

Two rendering paths

  • Legacy: DashboardConfig → resolveWidgetProps() → individual widget components
  • Enhanced: EnhancedDashboardConfig → <phz-widget> handles everything including processWidgetData()
  • engine-admin creates/edits the config objects; the host passes them to <phz-dashboard>

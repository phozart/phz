Defining Filters (Configuration)

Config

There are two ways to define filters: Legacy Mode (simple config object) and Registry Mode (enterprise engine with reusable definitions).

Legacy Mode (Simple)

Pass a CriteriaConfig object with an array of fields. Each field is a self-contained definition.

const config: CriteriaConfig = { fields: [ { id: 'region', label: 'Region', type: 'multi_select', dataField: 'region_code', required: true, options: [ { value: 'US', label: 'United States' }, { value: 'UK', label: 'United Kingdom' }, ], barConfig: { pinnedToBar: true, defaultOpen: true, }, }, { id: 'salary', label: 'Salary Range', type: 'numeric_range', numericRangeConfig: { min: 0, max: 300000, step: 5000, unit: 'USD', showSlider: true, }, }, ], };

Registry Mode (Enterprise)

Define filters once in a FilterRegistry, then bind them to reports/dashboards via FilterBindings.

import { createCriteriaEngine } from '@phozart/phz-engine'; const engine = createCriteriaEngine(); // Define once, reuse across artefacts engine.registry.register({ id: filterDefinitionId('region'), label: 'Region', type: 'multi_select', sessionBehavior: 'persist', dataField: 'region_code', options: [...], createdAt: Date.now(), updatedAt: Date.now(), }); // Bind to a specific report engine.bindings.bind({ filterDefinitionId: filterDefinitionId('region'), artefactId: artefactId('sales-report'), visible: true, order: 0, });

Selection Mode

Every field supports a selectionMode property that controls how many values can be selected:

ModeBehaviourApplies To
multiple Default for multi_select, chip_group, tree_select. Users can pick many values. multi_select, chip_group, tree_select
single Radio behaviour — selecting one deselects the previous. Tree hides "Select All". All list-based types
none Read-only display. Control is disabled, validation skips the field. All types

Per-Field Bar Config

Each field has an optional barConfig object that controls its bar/drawer appearance:

PropertyTypeDescription
pinnedToBarbooleanShow this field's value as a tag in the criteria bar
defaultOpenbooleanStart this section expanded in the drawer
expandablebooleanWhether the section can be collapsed (default true)
maxVisibleItemsnumberTruncate display after N items (0 = no limit)
enableMatchFilterbooleanEnable tri-state match filter pill (tree selects)
showOnSummarybooleanInclude in summary bar when barMode is 'summary'

External Data Sources for Options

Instead of hardcoding options, you can pull values from a DataSet at runtime:

{ id: 'region', type: 'multi_select', optionsSource: { dataSetId: 'regions-master', // key in dataSources map valueField: 'code', // column for option value labelField: 'name', // column for option label sortBy: 'label', // 'label' | 'value' | 'none' }, }

Filter Registry
filter-registry.ts

Central store for FilterDefinition objects. Artefact-independent: define once, bind to many reports.

  • register(def) Add a new definition
  • get(id) Retrieve by ID
  • update(id, patch) Partial update
  • deprecate(id) Soft-delete
  • validateDependencyGraph() DAG cycle detection
🔗
Filter Bindings
filter-bindings.ts

Associates definitions to artefacts (reports, dashboards) with per-binding overrides for label, defaults, order, visibility.

  • bind(binding) Associate def to artefact
  • unbind(defId, artId) Remove association
  • reorderBindings(artId, ids) Change filter order
  • resolveArtefactFields() Merge def + binding → SelectionFieldDef[]
💾
State Manager
filter-state.ts

6-level state resolution chain. Supports session persistence via pluggable StateStorageAdapter.

Resolution priority:
1. Rule values → 2. Preset → 3. Persisted → 4. Binding default → 5. Definition default → 6. All selected (null)
⚠ null means "all selected" (no filter), not "nothing selected".
⚙
Rules Engine
filter-rules.ts

5 rule types that programmatically constrain which options are available:

exclude_patternRegex to hide matching options
include_patternRegex to keep only matching options
value_setExplicit include/exclude list
tree_group_compareFilter tree nodes by group field
customPlug in your own evaluator function
📤
Criteria Output
criteria-output.ts

Produces structured ArtefactCriteria with typed operators, ready for query building.

11 operators: in, not_in, between, equals, not_equals, like, not_like, is_null, is_not_null, greater_than, less_than

Includes debounced subscription model for real-time filter updates.

🔒
Admin Service
filter-admin.ts

Permission-gated CRUD with immutability rules and audit logging.

  • createDefinition() With permission check
  • updateDefinition() Type locked if bindings exist
  • deprecateDefinition() Soft-delete with audit
  • getAuditLog() Full operation history

Quick Start: Full Round-Trip

import { createCriteriaEngine, filterDefinitionId, artefactId } from '@phozart/phz-engine'; // 1. Create the engine const engine = createCriteriaEngine({ debounceMs: 300 }); // 2. Register filter definitions engine.registry.register({ id: filterDefinitionId('region'), label: 'Region', type: 'multi_select', sessionBehavior: 'persist', options: [{ value: 'US', label: 'US' }, { value: 'UK', label: 'UK' }], createdAt: Date.now(), updatedAt: Date.now(), }); // 3. Bind to artefact const reportId = artefactId('sales-report'); engine.bindings.bind({ filterDefinitionId: filterDefinitionId('region'), artefactId: reportId, visible: true, order: 0, }); // 4. Resolve fields for rendering const fields = engine.resolveFields(reportId); // 5. Build structured criteria from user selections const criteria = engine.buildCriteria(reportId, { region: ['US'] }); // { artefactId, filters: [{ operator: 'in', value: ['US'], dataField: 'region_code', ... }] } // 6. Subscribe to changes (debounced) engine.subscribe(criteria => { console.log('Filters changed:', criteria.filters); });
⚡ Migration from legacy: Use migrateFromCriteriaConfig(config, artefactId) to convert an existing CriteriaConfig into a full criteria engine — zero code changes to your field definitions.
